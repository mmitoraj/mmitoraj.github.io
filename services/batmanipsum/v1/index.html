API Reference

/authorize

The OAuth2 Authorization Endpoint is used to authenticate and obtain an authorization grant.
For more information, see the OAuth 2.0 specification (rfc6749#section-3.1) and Getting Started tutorial.

/authorize

get

Starts the authorization procedure with the parameters specified in the URL query.

post

This starts the authorization procedure with the parameters specified in the request body (form parameters).

get /authorize

Starts the authorization procedure with the parameters specified in the URL query.

  • Request
  • Response

Query Parameters

  • response_type: required (string)

    This value can be "token" or "id_token token" (currently only the Implicit Grant flow, as described in rfc6749#section-4.2.1, is supported).
    For more information, see the OAuth 2.0 specification (rfc6749#section-3.1.1).

  • client_id: required (string)

    This is the client identifier, available in the Client Authorization section in the Builder. For more information, see the OAuth 2.0 specification (rfc6749#section-2.2).

  • redirect_uri: (string)

    This URL is used as a callback after a successful authentication and authorization grant to redirect the access token. This is:

    • OPTIONAL if only one redirect URI is registered for your application.
    • REQUIRED if multiple redirect URIs are registered for your application (must match at least one of these URIs).
      For security reasons, TLS should be used for redirect URIs.
      For more information, see the OAuth 2.0 specification (rfc6749#section-3.1.2).
  • scope: (string)

    This lists the requested scope of the access.
    For more information, see the OAuth 2.0 specification (rfc6749#section-3.3).
    The predefined scopes are:

    • hybris.tenant={tenantName} - This enables you to specify the tenant for which an access token should be issued.
      If neither this scope nor hybris.no_tenant is specified, and the user account is associated with multiple tenants, the authorization server displays an additional page which enables the user to select the appropriate tenant.
      If both this scope and hybris.no_tenant are specified, the server responds with a redirection to the redirect_uri or registered redirect URL with the 'error' parameter set to 'invalid_scope' and 'error_description' with the error details.
    • hybris.no_tenant - This enables you to omit the tenant chooser.
      If neither this scope nor hybris.tenant is specified, and the user account is associated with multiple tenants, the authorization server displays an additional page which enables the user to select the appropriate tenant.
      If both this scope and hybris.tenant are specified, the server responds with a redirection to the redirect_uri or registered redirect URL with the 'error' parameter set to 'invalid_scope' and 'error_description' with the error details.
    • hybris.org={organizationID} - This enables yout to specify the organization for which an access token should be issued.
      Using both this scope and hybris.tenant or hybris.no_tenant is not allowed.

    Example:

    hybris.tenant%3Dfa4dvht53fgb%20hybris.manage%20hybris.view
  • state: (string)

    RECOMMENDED. This is an opaque value used by the client to maintain the state between the request and the redirect.
    This value is included when redirecting the user agent back to the client.
    The parameter should be used to prevent cross-site request forgery as described in the OAuth 2.0 specification (rfc6749#section-10.12).

    Example:

    security_token%3D9F2a3290Daa126P%26custom_state%3Dhttps://hybris.com/index.html
  • id_token_hint: (string)

    This value is used in subsequent requests for the access token (the actual ID token that was received in the previous request).

  • nonce: (string)

    This value is used to prevent replay attacks when using id_token. It must have sufficient entropy.

  • display: (string)

    This value specifies how the authorization server displays the authentication and tenant chooser pages to the end user.
    The supported values are:

    • popup for transparent background
    • page for default display

    Example:

    page
  • hybris-session-id: (string)

    This is the session identifier. It is generated if not passed.

HTTP status code 200

The authentication page or the tenant chooser page for multi-tenant clients.

HTTP status code 302

The redirection to the redirect_uri or the registered redirect URL.

Headers

  • location: required (string)

    The redirection URL equals the redirect_uri if it was specified in the request. Otherwise, it equals the registered redirect URL. If multiple redirect URLs are registered, the redirect_uri parameter is required.
    It contains the following parameters in the URL Fragment Component:

    • If the authentication and the authorization grant succeed:
      • access_token - This is the access token issued by the authorization server.
      • token_type - This is the type of the token issued. The value is "bearer".
        For more information, see the OAuth 2.0 specification (rfc6749#section-7.1).
      • expires_in - This is the lifetime in seconds of the access token.
      • scope - This lists the effective scope of the access token. The value of the scope parameter is expressed as a list of space-delimited, case-sensitive strings.
        For more information, see the OAuth 2.0 specification (rfc6749#section-3.3).
      • state - This is the exact value received from the client if it was specified in the request "state" parameter.
    • If the resource owner denies the access request, or if the request fails for reasons other than a missing or invalid redirection URI:
      • error - This is a single ASCII error code from the following:
        • invalid_request - The request is missing a required parameter, includes an invalid parameter value, includes a parameter more than once, or is otherwise malformed.
        • unauthorized_client - The client is not authorized to request an access token using this method.
        • access_denied - The resource owner or authorization server denied the request.
        • unsupported_response_type - The authorization server does not support obtaining an access token using this method.
        • invalid_scope - The requested scope is invalid, unknown, or malformed.
        • server_error - The authorization server encountered an unexpected condition that prevented it from completing the request.
        • temporarily_unavailable - The authorization server is currently unable to handle the request due to a temporary overloading or maintenance of the server.
      • error_description - This is a human-readable ASCII text line providing additional information that is used to assist the client developer in understanding the returned error.
      • error_uri - This is a URI identifying a human-readable webpage with information about the error that is used to provide the client developer with additional information about the error.
      • state - This is the exact value received from the client if it was specified in the request "state" parameter.
      For more information, see the OAuth 2.0 specification (rfc6749#section-4.2.2.1).

HTTP status code 400

Bad request is returned if the parameter:

  • client_id is missing (the redirect_uri cannot be verified and redirect URL is not found)
  • client_id has an invalid value (the redirect_uri cannot be verified and redirect URL is not found)
  • client_id is included in the request more than once
  • redirect_uri is missing and is required because multiple redirect URLs are registered
  • redirect_uri has invalid value
  • redirect_uri is included in the request more than once
For more information, see the OAuth 2.0 specification (rfc6749#section-4.2.2.1).

/token

post

The token endpoint. Use only one of the authorization types: header, or client_id and client_secret parameters. For more information, see the OAuth 2.0 specification (rfc6749#section-3.2) and Getting Started tutorial.

post /token

The token endpoint. Use only one of the authorization types: header, or client_id and client_secret parameters. For more information, see the OAuth 2.0 specification (rfc6749#section-3.2) and Getting Started tutorial.

  • Request
  • Response

Headers

  • hybris-tenant: (string - minLength: 3 - maxLength: 16 - pattern: ^[a-z][a-z0-9]+$)

    The tenant that the caller is acting upon.

    Please note that this header's value is always lowercase.

    Example:

    myexampleshop
  • hybris-client: (string - minLength: 6 - maxLength: 41 - pattern: ^[a-z][a-z0-9-]{1,14}[a-z0-9][.][a-z][a-z0-9-]{0,22}[a-z0-9]$ )

    The identifier of the caller that is making the request.

    For the purpose of the request, the client is said to be a trusted client, if, and only if, the hybris-user header is absent. Please note that both dot-separated parts of this header's value are always lowercase.

    Example:

    myexample.myawesomeclient
  • hybris-client-id: (string)

    The hybris-client-id used to acquire the currently used access token. Providing the option to check if the current client id is indeed allowed to use the resource.

    Example:

    C0Kl5Sta1BMiQSqBQyYL5c0yRAwIr94a
  • hybris-external-path: (string)

    The public base path used by the calling client to make the request.

    Example:

    /hybris/echo/v1
  • hybris-external-url: (string)

    The public base URL used by the calling client to make the request.

    Example:

    https://api.yaas.io/hybris/echo/v1/
  • hybris-user: (string)

    The identifier of the authenticated user.

    This is the user that obtained the OAuth 2.0 access token. In YaaS, the user identifier usually is an email address.

    If this header is absent, but the hybris-client header is set, the request is said to be triggered by a trusted client. That is a service that authenticates itself, rather than leaving authentication to a user.

    Example:

    mustermann@hybris.com
  • hybris-scopes: (string - pattern: ^([a-zA-Z0-9._=-]{1,128}( [a-zA-Z0-9._=-]{1,128})*)?$)

    List of authorization scopes that are granted to the incoming request.

    Distinct scopes are separated by single space characters and may be listed in any order. Each scope has a maximum length of 128 characters, and it typically is of the form <org_base_path>.<service_name>_<scope_name>. Please note that scopes are case sensitive.

    This header may be empty or absent, e.g. when the request does not require any authorization scopes, or the caller cannot obtain any.

    Example:

    my-org.myService_view my-org.myService_manage
  • hybris-request-id: (string)

    Unique identifier of the current HTTP request chain. The service propagates it unaltered whenever it makes a subsequent request to another YaaS service.

    Example:

    Ahd75sgAf54d
  • hybris-hop: (integer - default: 1 - minimum: 1)

    The level of depth in the current HTTP request chain. The service propagates it unaltered whenever it makes a subsequent request to another YaaS service.

    Example:

    3
  • hybris-org: (string)

    The identifier of the organization the caller is acting upon.

    Example:

    59cc4b46f5ce83bf493f7h29
  • hybris-target-url: (string)

    The full public URL used by the calling client to make the request

    Example:

    https://api.yaas.io/hybris/echo/v1/myresource/1?fields=id
  • Authorization: (string)

    Provides the basic authorization. For more information, see the OAuth 2.0 specification (rfc6749#section-2.3.1).

    Example:

    Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW

Body

Type: application/x-www-form-urlencoded

Form Parameters
  • grant_type: required (string - default: client_credentials)

    The grant type of the request. The value MUST be set to client_credentials.

    Example:

    client_credentials
  • scope: (string)

    Lists the requested scope of the access.
    For more information, see the OAuth 2.0 specification (rfc6749#section-3.3).
    The predefined scope is:

    • hybris.tenant={tenantName} - This enables you to specify the tenant for which an access token should be issued.
      It may be used for multi-tenant clients: Services or Builder modules. It is set to the ID of the team that created the client if not passed. This must be set to the ID of the project that should be used in subsequent requests with the access token. For single-tenant clients when the client_id of the project is used, do not set this parameter.

    Example:

    hybris.tenant=fa4dvht53fgb hybris.manage hybris.view
  • client_id: (string)

    Client identifier. This is required if the Authorization header is omitted.
    For more information, see the OAuth 2.0 specification (rfc6749#section-2.2).

  • client_secret: (string)

    Client secret. This is required if the Authorization header is omitted.
    For more information, see the OAuth 2.0 specification (rfc6749#section-2.3.1).

  • hybris-session-id: (string)

    Session identifier. It is generated if not passed.
